## Scope decisions (PLAN.md is terse / ARCHITECTURE.md's aspect table is broader than PLAN's slice — recording gap-fills before coding)

1. **`tickets.chatbot_session_id` is a plain nullable column, not an FK yet.** `chatbot_sessions` doesn't exist until P11. Same forward-reference pattern as P1's reservation stub / P9's undefined P10 triggers: the column exists (matches §7's schema) but is unconstrained until P11 adds the real table + relation. `conversation_id` **is** a real FK — `conversations` exists since P9.
2. **`tickets.source` added** (not in §7's abbreviated schema, but named explicitly in §5.11 prose: `"Ticket with source = chatbot"`). String column, default `chatbot` (the only source that exists until a human-created-ticket path is ever specified) — cheap to add now, avoids a P11 remediation migration for a field already named in the architecture doc.
3. **Ticket category → department is a new mapping** (PLAN doesn't specify one; §5.11's category enum — `inquiry|complaint|booking_help|maintenance|other` — doesn't share vocabulary with `ServiceRequest`'s room-service/housekeeping types). Judgment call, documented on `Ticket::CATEGORY_DEPARTMENTS`: `complaint`→concierge, `maintenance`→housekeeping, `booking_help`→reception, `inquiry`/`other`→concierge (same default-to-concierge fallback pattern as `ServiceRequest::TYPE_DEPARTMENTS`).
4. **`RouteRequestAction` has no HTTP route in P10 — same treatment as P4's `ReleaseExpiredHoldsAction`.** PLAN's route list for this phase is exactly `GET /operations/queue`, assign/status (polymorphic `{type}`), `GET /dashboard/summary` — no "re-route" endpoint. Nothing in P10 itself creates a `Ticket` (P11's `CreateTicketAction` does), so the action's real caller doesn't exist yet either. Built and unit-tested directly (`app(RouteRequestAction::class)->handle(...)`) against **both** `ServiceRequest` and `Ticket` — satisfying "shared by service_requests and tickets" — ready for P11 to call at ticket-creation time. P6/P7's own working, tested inline routing (`SubmitInquiryAction`, `PlaceServiceRequestAction`) is left untouched — refactoring shipped code to delegate to this new shared primitive isn't requested and isn't free of regression risk for zero behavior change.
5. **Unified queue is a true merge, per PLAN's own test bullet ("unified queue merges requests + tickets").** `OperationsQueueService::index()` pulls both querysets (each filtered to only the tables the caller holds `*.view` for — holding neither returns an empty page, not a 403; the route itself requires `service_requests.view|tickets.view` so at least one is guaranteed), normalizes both into one shape via `OperationsQueueItemResource` (`type` discriminator + shared display fields + item-specific nested payload), sorts by `created_at` desc, and hand-paginates the merged collection.
6. **`GET /dashboard/summary` scope is slightly wider than the queue's "both tables."** ARCHITECTURE §5.7 lists the dashboard's read scope as Events + Pre-Arrival + In-room + Tickets — broader than PLAN's queue-specific "both tables." Since summary is read-only aggregate counts (no new mutation surface), it includes a third block — event-inquiry counts by status — gated by the same `tickets.view`/`tickets.assign` permissions P6 already used for event-inquiry admin routes. Pre-arrival (`check_in_approvals`) is left out: PLAN doesn't name it and it has no natural "queue" shape (a boolean approval per reservation, not a status-bearing ticket). Service-request and ticket blocks are gated by their own `.view` permissions, independently.
7. **Assign/status endpoints are one polymorphic route pattern (`/operations/queue/{type}/{uuid}/...`) with the permission check done inside the service, not route middleware.** `service_requests` and `tickets` need *different* permissions per operation (`service_requests.assign` vs `tickets.assign`; `service_requests.update` vs `tickets.respond` — reusing the same chat-reply permission from P9, matching that precedent), which static `permission:` route middleware can't branch on a URL segment. `OperationsQueueService` resolves the model by `{type}` and throws `ForbiddenException` if the caller lacks the type-appropriate permission.
8. **Ticket status has no guarded state machine** (unlike `EventInquiry`/`Reservation`). PLAN's done-condition doesn't ask for transition guards, only that status updates happen and mirror live; validation is simply "one of the model's defined status values" via the request rule.

## Slice
- **Migration:** `tickets` (uuid, guest_id? FK, chatbot_session_id? plain nullable column, conversation_id? FK→conversations nullOnDelete, subject, category, status default `open`, priority int default 2, department, source default `chatbot`, assigned_user_id? FK→users nullOnDelete).
- **Model:** `Ticket` — status/category/source constants, `CATEGORY_DEPARTMENTS` map, `guest()`/`conversation()`/`assignedUser()` relations.
- **Actions:** `RouteRequestAction` (department resolver, `ServiceRequest|Ticket` — no route, unit-tested), `AssignRequestAction`, `UpdateRequestStatusAction` (both `ServiceRequest|Ticket`, mirror to Firestore via the P9 `MirrorsToFirestore` trait, same `ops_queue` collection `MirrorServiceRequestToFirestore` already writes to).
- **Service:** `OperationsQueueService` — `index()` (merged, permission-filtered, paginated), `summary()` (permission-filtered counts), `assign()`, `updateStatus()` (both resolve-by-type + permission-check + delegate to the Actions above).
- **Resource:** `OperationsQueueItemResource` — normalizes `ServiceRequest`/`Ticket` to one shape.
- **Routes (`auth:users`):** `GET /operations/queue` (`service_requests.view|tickets.view`), `PATCH /operations/queue/{type}/{uuid}/assign`, `PATCH /operations/queue/{type}/{uuid}/status` (permission checked in-service per type), `GET /dashboard/summary` (any authenticated staff — the service itself filters each block by permission, so a user with nothing returns an all-null/empty summary rather than a 403).
- **Factory:** `TicketFactory`.
- **Tests:** category/type → department routing (both models, direct Action call); unified queue merges + sorts across both tables and excludes a table the caller can't view; assign updates `assigned_user_id` + mirrors to Firestore (fake); status update validates against the model's own enum + mirrors; summary respects permissions (block present/absent per what the caller holds); permission gates on assign/status per type.

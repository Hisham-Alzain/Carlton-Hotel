Real linking is built once, correctly, in **P4.R**.

## R2 — Test the OTP hourly cap (5/hr)

The 5/hour per-`(identifier, purpose)` cap (`RequestOtpAction`, key `otp:hour:{identifier}:{purpose}`, limit 5/3600s) was never tested in P1. It now bounds soft-hold creation under the P4 flow, so it protects inventory. Add the test.

### R2.1 — Create `OtpHourlyCapTest`

**File:** `tests/Feature/Auth/OtpHourlyCapTest.php` (new)

- Namespace: `Tests\Feature\Auth`
- Class: `OtpHourlyCapTest extends Tests\TestCase`, `use RefreshDatabase;`
- In `setUp()`, call `parent::setUp();` then `OtpDispatcher::reset();` (mirror `BookingCodeLinkTest`).

**How the two gates interact:** the 1/min gate (`otp:min:{identifier}:{purpose}`, limit 1/60s) fires before the hourly gate. To exercise the 5/hr cap you must get past the minute gate on each of the first 5 requests. The cleanest way is **time travel**: place each request in a distinct minute inside the same hour, so the minute limiter's 60s decay clears between hits while the hour limiter (3600s decay) keeps accumulating. Use `$this->travel(N)->minutes()` (or `Carbon::setTestNow`) and end with `$this->travelBack();`.

**Method 1 — `test_sixth_request_within_hour_is_rate_limited` (purpose = login):**

- Pick one identifier, e.g. `$phone = '+96393333333';`
- Clear both limiter keys up front:
  ```php
  RateLimiter::clear("otp:min:{$phone}:login");
  RateLimiter::clear("otp:hour:{$phone}:login");
  ```
- Loop the first 5 requests, advancing time by 1 minute before each so the minute gate is clear each time:
  ```php
  for ($i = 0; $i < 5; $i++) {
      $this->travel($i)->minutes(); // 0,1,2,3,4 min — same hour
      $this->postJson('/api/auth/guest/request-otp', [
          'phone' => $phone, 'channel' => 'sms', 'purpose' => 'login',
      ])->assertStatus(200);
      $this->travelBack();
  }
  ```
  (Advance from a fixed base with `Carbon::setTestNow` if `travel()` accumulation is awkward; the requirement is: 5 successes inside one 60-minute window, each clear of the 1/min gate.)
- 6th request, still inside the hour, must be capped:
  ```php
  $this->travel(5)->minutes();
  $this->postJson('/api/auth/guest/request-otp', [
      'phone' => $phone, 'channel' => 'sms', 'purpose' => 'login',
  ])->assertStatus(429)->assertJson(['error_code' => 'too_many_requests']);
  $this->travelBack();
  ```
- Assert the 6th response is `429` with `error_code = too_many_requests`.

**Method 2 — `test_hourly_cap_applies_to_booking_verification_purpose` (purpose = booking_verification):**

- Identical structure to Method 1, but with `'purpose' => 'booking_verification'` on every request and limiter keys `otp:min:{$phone}:booking_verification` / `otp:hour:{$phone}:booking_verification`.
- Use a **different identifier** than Method 1 (e.g. `+96393333334`) to avoid any cross-test limiter bleed.
- **Note:** `RequestOtpRequest` (P1) validates `purpose` as `required|in:login,register`. If the request-layer validation rejects `booking_verification` with a 422, this method must instead exercise the action/limiter directly rather than the HTTP endpoint: construct the identifier and call `app(\App\Actions\Auth\RequestOtpAction::class)->handle($phone, 'sms', 'booking_verification')` 5× (with the same time-travel minute spacing), then assert the 6th call throws `App\Exceptions\TooManyRequestsException`. This keeps the test forward-compatible with P4 without loosening P1 request validation in this phase. Pick whichever path matches current validation; prefer the action-level assertion if the endpoint 422s on the new purpose.
- Assert the same 5-pass / 6th-blocked behaviour holds for `booking_verification`.

- Imports: `use Illuminate\Foundation\Testing\RefreshDatabase;`, `use Illuminate\Support\Facades\RateLimiter;`, `use App\Actions\Auth\OtpDispatcher;`, `use Tests\TestCase;` (and `App\Exceptions\TooManyRequestsException` if using the action-level path in Method 2).

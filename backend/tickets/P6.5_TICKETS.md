## R1 — Add `is_checked_in` to the guest entitlement payload (second flag)

**Problem:** ARCHITECTURE §3.7 (revision 4) splits the old single `has_active_reservation` into two flags — `has_booking` (pre-arrival tier) and `is_checked_in` (in-room tier) — because P7's gate needs to distinguish "booked" from "physically checked in."

**Inspection finding (read before coding):** `GET /auth/me` for guests **does not exist** in the current codebase. Only `GET /auth/me` for staff exists (`StaffAuthController::me`, `auth:users` guard). `has_active_reservation` currently only appears embedded in `GuestResource`, which is returned from `POST /auth/guest/verify-otp` — and that action does **not** eager-load `activeReservations`, so the flag is always `false` there today. PLAN.md's phrasing ("P4 shipped `GET /auth/me` with a single flag") does not match what was actually built; treat this ticket as **build the endpoint correctly now**, not "add a second flag to an existing one."

**Scope:**
1. Add `GET /api/auth/guest/me` (new route, `auth:guests` guard, symmetric with the existing `/auth/guest/*` routes — does not collide with staff's `/auth/me`).
2. `AuthGuestService::me(Guest $guest): array` — eager-loads `activeReservations`, returns `['data' => $guest, 'code' => 200]`.
3. `GuestAuthController::me()` — resolves `$request->user('guests')`, calls service, wraps in `GuestResource`.
4. `GuestResource` — compute from the loaded `activeReservations` collection (no new query, per convention — Resources never query):
   - `has_booking`: true if any loaded reservation has `status` in `[confirmed, checked_in]`.
   - `is_checked_in`: true if any loaded reservation has `status = checked_in`.
   - `has_active_reservation`: **kept as a deprecated alias, equal to `has_booking`** — `API_GUIDE_MOBILE.md` documents the app reading this field today (tier table + two other references), so it cannot be silently renamed. Deprecation to be revisited in a later phase once the app ships on the new fields.
   - `active_reservation`: summary now reflects the confirmed/checked-in reservation (not just "any non-cancelled/non-checked_out" reservation as before) — tightens correctness to match the two new flags.
5. Add `ReservationFactory::checkedIn()` state (missing — only `confirmed`, `cancelled`, `pendingVerification`, `expiredHold` exist).
6. Update `API_GUIDE_MOBILE.md`: document the real route (`GET /api/auth/guest/me`), both new flags, which tier each unlocks, and the alias's deprecation note.

**Test (`tests/Feature/Auth/GuestMeTest.php`):**
- No reservation → `has_booking=false, is_checked_in=false, has_active_reservation=false`.
- Confirmed (not checked in) reservation → `has_booking=true, is_checked_in=false`.
- Checked-in reservation → `has_booking=true, is_checked_in=true`.
- `has_active_reservation` always equals `has_booking` (alias assertion).
- Unauthenticated request → 401.

**Record in LOG:** the missing-endpoint finding above, and the alias decision (kept, not renamed).

## R2 — Give P6 event-inquiry state errors a distinct `error_code`

**Problem:** `EventInquiryService::updateStatus()` currently throws `ReservationStateException` (`error_code: reservation_state`) for an invalid inquiry status transition. P10 will need to distinguish inquiry-state errors from reservation-state errors on the frontend.

**Scope:**
1. `app/Exceptions/InquiryStateException.php` — extends `DomainException`, `errorCode(): 'inquiry_state'`, `statusCode(): 422`.
2. `EventInquiryService::updateStatus()` — throw `InquiryStateException` instead of `ReservationStateException`. Same message key (`custom.errors.inquiry_state` — already exists in en+ar from P6, no lang change needed).
3. Update `tests/Feature/Events/EventInquiryTest::test_invalid_status_transition_returns_422` to assert `error_code === 'inquiry_state'`, not just the 422 status.

**Record in LOG:** confirm the lang key already existed (added in P6) and required no change — only the exception class + throw site changed.
